## 背景

微信小程序架构采用模块化单体，数据层以 MySQL 8.0 为业务主库，配合 Redis、OSS 和后续异步基础设施。当前架构文档已经识别商品浏览、购物车持久化、下单、微信支付回调、库存锁定、优惠券使用、售后处理、运营管理和领域事件等核心数据流。

本变更将架构草图收敛为数据库设计契约。表结构需要支撑 MVP 上线，同时为后续读写分离、事件 worker、归档表和模块拆分预留空间。

关键约束：

- MySQL 8.0 是业务记录的事实源。
- Redis 仍承担购物车和库存的热路径，但 MySQL 保存持久化快照和对账数据。
- 所有表均不包含 `org_id`。
- 所有表都保留记录，业务流程必须通过状态字段表达失效、关闭、过期、取消、禁用或归档标记，不删除数据。
- 不使用数据库外键约束，表关系通过文档说明并由应用服务层维护。

## 目标 / 非目标

**目标：**

- 定义电商小程序 MVP 及运营管理端所需的完整表集合。
- 统一所有表的基础字段、命名、状态化生命周期和审计规则。
- 为小程序端、运营管理端、定时任务和 worker 的常见查询路径设计索引。
- 保留业务历史，支持审计、对账、客服、支付争议处理和运营分析。
- 保持设计与 Go GORM 模型、Repository 和 Service 事务边界兼容。
- 金额、价格、折扣、运费、退款等金额类字段统一使用 `DECIMAL(19,7)`。

**非目标：**

- 本提案不实现迁移脚本、GORM 模型、Repository 或 API Handler。
- 不引入多租户 `org_id` 字段。
- 不添加数据库级外键约束。
- 不定义 Redis Lua 脚本或完整异步 worker 实现细节。
- MVP 不实施分库分表，仅保留不阻碍后续分片的键和索引选择。

## 决策

### 决策 1：使用覆盖所有领域的完整 MVP 表结构

本设计覆盖用户、地址、品牌、分类、商品、SKU、购物车、订单、订单明细、订单日志、支付、退款、库存、库存流水、优惠券、用户优惠券、秒杀活动、售后单、发货、运营管理表、领域事件和幂等记录。

原因：现有架构已经采用模块化单体。完整表结构可以避免各模块自行发明生命周期和索引约定。

备选方案：只先定义订单、支付、库存表。该方案被拒绝，因为用户、商品、购物车、营销、售后和系统支撑表都会影响下单与对账链路。

### 决策 1.1：包含标准运营管理端表集合

运营管理端采用常见后台系统表结构：

- 账号与授权：`admin_users`、`admin_roles`、`admin_permissions`、`admin_user_roles`、`admin_role_permissions`。
- 控制台导航：`admin_menus`。
- 安全与审计：`admin_login_logs`、`admin_operation_logs`。
- 平台配置：`system_configs`。
- 图片与内容运营：`image_assets`、`operation_banners`、`operation_nav_items`、`file_assets`。
- 履约配置：`logistics_companies`、`freight_templates`、`freight_template_rules`。

原因：这些表符合常见电商运营后台做法，同时对 MVP 足够轻量。它们支持账号控制、RBAC、菜单渲染、操作追踪、首页运营、素材复用和运费管理，避免把运营配置混入交易表。

备选方案：MVP 仅保留 `admin_users`。该方案被拒绝，因为商品、订单、退款、营销和首页运营从一开始就需要权限控制和审计日志。

### 决策 1.2：通过独立图片表管理 OSS 图片

图片通过 `image_assets` 表管理，而不是只在业务表中保存裸 URL。该表记录 OSS bucket、object key、公开 URL 或 CDN URL、原始文件名、内容类型、文件大小、宽度、高度、checksum、图片用途、上传人、状态和可选元数据。

常见图片用途包括商品主图、商品图集、SKU 图、分类图标、品牌 Logo、小程序首页轮播图、导航图标、营销活动图、售后凭证图和富文本内嵌图。

`operation_banners` 引用 `image_assets.id` 作为轮播图图片，并保存跳转类型、跳转目标、展示时间窗口、排序和状态。商品、分类、品牌、营销等表可以对主图引用 `image_assets.id`，也可以在需要保序的图集中使用 JSON 数组保存图片资产 ID。

原因：OSS 负责真实二进制对象存储，MySQL 保存元数据、用途、状态和可查询管理字段。这样可以支持图片复用、状态控制、审计、CDN 替换、图片清理检查和运营后台检索。

备选方案：直接在商品、分类和轮播图表中保存图片 URL。该方案被拒绝，因为仅保存 URL 很难管理 OSS 对象生命周期、去重上传、审计使用情况和全局禁用问题图片。

### 决策 1.3：金额类字段统一使用 `DECIMAL(19,7)`

所有金额、价格、折扣、运费、退款、优惠门槛、优惠金额、秒杀价等涉及金额或价格的小数字段统一使用 `DECIMAL(19,7)`。

原因：电商业务后续可能涉及精细折扣、促销试算、分摊、退款拆分、积分/余额兑换或第三方结算，`DECIMAL(10,2)` 精度不足。统一 `DECIMAL(19,7)` 可以避免不同表之间精度不一致导致的舍入误差和对账差异。

备选方案：使用整数分或 `DECIMAL(10,2)`。整数分对 7 位小数诉求不友好，`DECIMAL(10,2)` 无法满足至少 7 位小数的精度要求，因此均不采用。

### 决策 2：所有表保留记录并通过状态字段表达生命周期

每张表保留 `created_at`、`updated_at`、`deleted_at` 以保持模型一致性，但正常业务流程不得写入 `deleted_at`，也不得物理删除行。所有无效状态都通过明确状态字段表达，例如用户禁用、商品下架、订单取消、优惠券过期、事件失败或售后关闭。

原因：电商数据需要支持审计、支付对账、退款、客服和运营分析。全局不删除规则比按表设置例外更简单、更安全。

备选方案：允许非金融表软删除。该方案被拒绝，因为商品、购物车、优惠券、地址和后台账号数据都可能影响历史订单和客服排查。

### 决策 3：不包含 `org_id`

任何表都不包含 `org_id`，任何唯一索引也不包含 `org_id`。

原因：目标小程序是单一业务平台，不是多租户 SaaS。提前加入 `org_id` 会增加查询和唯一性约束复杂度，但当前没有明确需求。

备选方案：为未来多租户预置 `org_id`。该方案被拒绝，以保持 MVP 表结构和索引聚焦。

### 决策 4：不使用数据库外键约束

表结构会为关联字段建立索引并说明关系，但不声明数据库外键约束。

原因：Go 服务层负责跨模块事务和完整性规则。不使用数据库外键可以让批处理、异步 worker、导入和后续服务拆分的写入更简单。

备选方案：使用数据库外键保证强完整性。该方案被拒绝，因为现有架构明确倾向由应用层维护模块边界。

### 决策 5：围绕真实查询路径设计索引

索引覆盖：

- 业务唯一性：`openid`、`spu_code`、`sku_code`、`order_no`、`after_sale_no`、`idempotency_key`。
- 小程序端列表：用户订单列表、购物车列表、优惠券列表、地址列表。
- 运营后台扫描：管理员账号、角色编码、权限编码、菜单树、登录日志账号/时间、操作日志模块/时间、图片用途/状态/时间、商品状态、订单状态、支付状态、售后状态、事件重试状态。
- Worker 路径：超时关单、领域事件重试、库存对账、支付回调幂等。
- 游标分页：`created_at` 或业务时间字段配合状态/用户过滤。

原因：MVP 预期峰值 QPS 约 200，后续增长到 2000。聚焦复合索引可以在不过度索引的前提下提供稳定性能。

备选方案：为大多数字段创建单列索引。该方案被拒绝，因为主查询通常是用户/状态/时间或状态/时间组合条件。

## 风险 / 取舍

- 全局不删除策略会增加存储增长 -> 通过归档表、冷存储和后台筛选缓解，同时保留原始记录。
- `deleted_at` 存在但正常不写入 -> 通过代码审查规则和只暴露状态流转方法来避免误用删除 API。
- 不使用数据库外键可能产生孤儿引用 -> 通过服务层校验、事务边界、对账任务和关联字段索引缓解。
- 复合索引会增加写入成本 -> 只为明确的商品、订单、支付、营销、图片和 worker 查询路径建索引。
- JSON 快照字段可能变大 -> 只在历史不可变性重要的场景保存快照，如订单、购物车加购、售后申请和支付回调。
- 运营表可能演进成完整 IAM 系统 -> MVP 保持账号-角色-权限-菜单的表驱动模型，部门、岗位、审批流等后置。
- OSS 图片元数据可能与真实 OSS 对象漂移 -> 通过上传回调、checksum/object key 唯一性和定期对账任务标记缺失或无效对象。
- `DECIMAL(19,7)` 比 `DECIMAL(10,2)` 占用更多存储 -> 以统一精度和减少对账误差优先，避免后续迁移金额字段。

## 迁移计划

1. 创建数据库设计文档或迁移脚本，定义所有 MVP 表、字段注释、索引和状态枚举。
2. 生成或更新 GORM 模型，遵守本设计的字段和精度约定。
3. 围绕索引查询路径实现 Repository 方法。
4. 增加服务层保护，禁止物理删除和软删除。
5. 按需增加基础分类、后台初始账号、系统配置等初始化迁移。
6. 在本地 MySQL 8 实例执行迁移并检查索引创建情况。

回滚策略：由于这是初始表结构设计，非生产环境回滚可删除新建 MVP 表。生产环境回滚需要先停止写入并保留数据备份。
