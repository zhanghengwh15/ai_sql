## 为什么

现有小程序架构已经定义了业务模块和核心数据流，但数据库模型仍停留在概要层面。实施前需要形成一套完整的数据库设计提案，让用户、商品、购物车、订单、支付、库存、营销、售后、运营管理、图片管理和系统支撑表具备统一的生命周期、索引和审计规则。

## 变更内容

- 设计电商微信小程序 MVP 及运营管理端所需的完整 MySQL 8 表结构。
- 覆盖用户、地址、商品、SKU、购物车、订单、支付、退款、库存、优惠券、秒杀、售后、发货、图片管理、运营管理、幂等记录和领域事件数据。
- 增加市面上常见的运营管理端表：后台账号、RBAC 角色权限、菜单、登录日志、操作审计日志、系统配置、首页轮播图、导航入口、运费模板、物流公司、图片资源管理和文件素材管理。
- 增加基于 OSS 的图片管理表，用于小程序图片、商品图片、分类图标、首页轮播图、营销图片和运营后台可复用图片资源。
- 统一金额、价格、折扣、运费、退款等金额类字段精度为 `DECIMAL(19,7)`，至少保留 7 位小数。
- 建立全局“不删除”数据生命周期规则：所有表保留记录，通过状态字段表达禁用、关闭、取消、过期、失效等业务状态，不执行物理删除或逻辑删除。
- 所有表设计均不包含 `org_id`。
- 为业务唯一键、常用查询路径、状态扫描、游标分页、订单/支付幂等和异步事件重试设计索引。
- 不设置数据库外键约束，由应用服务层维护关联完整性。

## 能力

### 新增能力

- `miniapp-database-schema`：定义电商小程序平台及运营管理端所需的数据表、生命周期规则、表关系和索引要求。

### 修改能力

- 无。

## 影响范围

- 影响数据库迁移脚本、Go GORM 模型、Repository、Service 事务边界、小程序端 API 和运营管理端 API。
- 将 Redis 购物车和库存热路径与 MySQL 持久化表对齐。
- 为下单、支付回调、退款、库存对账、优惠券使用和领域事件 worker 提供数据契约。
- 为运营后台登录、RBAC 授权、菜单渲染、操作审计、首页运营、图片管理、物流配置和运费管理提供数据契约。
