# 电商微信小程序 · 详细架构设计

> 适用范围：微信小程序 · 电商/零售业务 · MVP 至 DAU 1 万级
> 设计目标：低成本快速上线、模块清晰可演进、关键链路稳定可观测
> 文档版本：v1.0 · 2026-05-10

---

## 1. 设计目标与约束

### 1.1 业务目标

构建一个面向 C 端消费者的微信小程序电商平台，覆盖商品浏览、下单、微信支付、订单履约、售后、会员与营销的完整闭环。MVP 阶段以「跑通主链路、控制成本、为后续放量留架构空间」为核心。

### 1.2 关键非功能性需求（NFR）

| 维度 | MVP 目标 | 12 个月演进目标 |
|---|---|---|
| 在线用户 | 峰值 QPS ≤ 200 | 峰值 QPS ≤ 2000 |
| 接口 P99 时延 | ≤ 500 ms | ≤ 300 ms |
| 可用性 | 99.5% | 99.9% |
| 首屏可交互（小程序） | ≤ 1.5 s | ≤ 1.0 s |
| 支付成功率 | ≥ 99.0% | ≥ 99.5% |
| RPO / RTO | RPO ≤ 1h / RTO ≤ 4h | RPO ≤ 5min / RTO ≤ 30min |

### 1.3 关键约束

- **平台约束**：微信小程序运行环境受限（包体 ≤ 2 MB 主包、≤ 20 MB 总包；接口域名需 ICP 备案 + HTTPS + 配置在白名单；只能调用微信能力开放清单内的 API）。
- **支付约束**：必须使用微信支付（JSAPI 方式），需要企业资质开通商户号。
- **资源约束**：MVP 阶段以 1 名前端 + 1 名后端 + 0.5 名运维的小团队为前提，架构必须**可由小团队独立维护**。
- **演进约束**：架构应允许在不大重构的前提下，平滑过渡到容器化、读写分离、消息队列异步化。

## 2. 整体架构总览

### 2.1 分层视图

整个系统采用**经典四层架构 + 模块化单体后端**，避免 MVP 阶段过早引入微服务带来的运维复杂度。

```
┌──────────────────────────────────────────────────────────┐
│                     接入层 (Access Layer)                  │
│  微信小程序  ·  H5 (备用)  ·  运营后台 (Web Admin)          │
└──────────────────────────────────────────────────────────┘
                             │ HTTPS
┌──────────────────────────────────────────────────────────┐
│                     网关层 (Gateway Layer)                 │
│   Nginx + WAF  ·  限流/鉴权/日志/灰度路由                   │
└──────────────────────────────────────────────────────────┘
                             │
┌──────────────────────────────────────────────────────────┐
│                   应用服务层 (Application)                  │
│   模块化单体 (Monolith-Modular)：                           │
│   用户 · 商品 · 购物车 · 订单 · 支付 · 营销 · 库存 · 售后    │
└──────────────────────────────────────────────────────────┘
                             │
┌──────────────────────────────────────────────────────────┐
│                     数据层 (Data Layer)                    │
│   MySQL 主库 · Redis 缓存 · 对象存储 OSS · ES (后期)         │
└──────────────────────────────────────────────────────────┘
                             │
┌──────────────────────────────────────────────────────────┐
│                  外部服务 (External Services)               │
│   微信开放平台 · 微信支付 · 短信 · 物流 API · 客服          │
└──────────────────────────────────────────────────────────┘
```

### 2.2 关键架构决策

| 决策项 | 选择 | 备选 | 取舍理由 |
|---|---|---|---|
| 后端架构形态 | 模块化单体 | 微服务 / Serverless | DAU < 1 万阶段，单体足够。模块边界清晰，可日后按模块拆分服务。 |
| 后端语言/框架 | **Go 1.22 + Gin** | Java/Spring Boot, Node.js/NestJS | 性能优异（单实例承载 QPS 是 Node 的 3~5 倍）、内存占用低、单二进制部署；Gin 轻量高性能，配合 Wire 做依赖注入即可达到工程规范性；面对秒杀类高并发场景天然友好。 |
| 主数据库 | **MySQL 8.0** | PostgreSQL | 生态成熟，云厂商一键部署，团队熟悉度高。 |
| 缓存 | **Redis 6.x** | Memcached | 数据结构丰富（库存、购物车、限流均依赖）。 |
| 对象存储 | **阿里云 OSS** | 腾讯云 COS, 七牛云 | 与阿里云 CDN/SLB/ECS 同 VPC 内网回源免流量费；签名直传成熟，小程序生态已封装好 SDK。 |
| 部署形态 | **2 台 ECS + RDS（MVP） → ACK（演进）** | 一开始就 ACK | MVP 不必要承受 K8s 运维成本，ECS + systemd 足够。 |
| 消息异步化 | **DB 消息表 + 定时器（MVP） → RocketMQ/Kafka（演进）** | 一开始就 MQ | 控制依赖数量，订单量小时 DB 表足够。 |

## 3. 前端架构（小程序端）

### 3.1 技术栈

- **框架**：**uni-app · Vue 3 + Vite + TypeScript**（多端一码：微信小程序为主，H5 备用入口，App 远期）
- **状态管理**：**Pinia**（Vue 3 官方推荐，轻量、类型友好、SSR/MP 通用）
- **UI 组件库**：**uv-ui** / `uni-ui` 官方组件，电商场景可叠加 `@uni-helper/vant-weapp` 或自封装业务组件库
- **网络层**：自研 `request<T>` 封装（基于 `uni.request`）—— 统一 baseURL、JWT 注入、traceId、401 自动续期、失败重试、loading
- **样式方案**：SCSS + **UnoCSS** 原子化（可选，减小体积、加速样式开发）
- **辅助工具**：`unplugin-auto-import` 自动引入 Vue/Pinia/uni API；`vue-tsc` 类型检查
- **构建/工程**：Vite（原生小程序通过 `@dcloudio/vite-plugin-uni` 编译）；`miniprogram-ci` 自动上传体验版；`eslint` + `prettier`
- **真机调试**：HBuilderX 或 CLI + 微信开发者工具二维码预览

> **选型理由**：
> - **多端覆盖**：当前主投微信小程序，H5 作为运营后台/外推链路备用入口，未来可能上 App，**uni-app 一份代码三端复用**省去重复实现成本。
> - **Vue 3 生态成熟**：组合式 API + TS + Pinia 类型推导，团队学习成本低。
> - **性能可控**：开启分包预加载 + tree-shaking + `lazyCodeLoading=requiredComponents`，能达到接近原生的体验。
> - **若团队后续完全聚焦微信单端、追求极致性能**，可在不重构页面层的前提下逐步切到 Skyline 或迁移核心页到原生（Vue → WXML 渐进式）。

### 3.3 分包策略（包体优化）

微信小程序主包上限 2 MB，必须分包；在 uni-app 中通过 `src/pages.json` 的 `subPackages` + `preloadRule` 配置：

- **主包**：tabBar 页面 + 登录/启动逻辑 + 公共组件 + 公共图片，控制在 1.5 MB 内
- **pages-goods**：商品详情、列表、搜索，预计 1.5 MB（含商品组件、SKU、富文本渲染）
- **pages-order**：订单全流程（确认、详情、列表、售后、物流），预计 1 MB
- **pages-marketing**：营销（秒杀、拼团、优惠券中心），预计 1 MB
- 启用 **分包预下载** `preloadRule`：进入首页后预下载 `goods`，进入商品详情后预下载 `order`
- `manifest.json` 中开启 `lazyCodeLoading=requiredComponents` + `optimization.subPackages=true` 进一步减包

> uni-app 编译产物会自动按主包/分包目录结构组织 WXML/WXSS/WXS。

### 3.4 状态管理与数据流（Pinia）

采用「**单向数据流 + 领域 Store**」：

- 每个领域一个 Pinia store：`useUserStore` / `useCartStore` / `useOrderStore` / `useMarketingStore`
- 页面通过 `storeToRefs(useXxxStore())` 解构响应式字段使用
- 跨页面共享数据（用户信息、购物车数量徽标）从 store 读取，**不依赖页面间参数传递**
- 持久化：登录态、购物车快照走 `uni.setStorageSync`（自定义 Pinia 持久化插件，按 store 配置白名单字段），启动时自动 hydrate

### 3.5 网络层关键设计

- **统一封装** `request<T>(options)`（基于 `uni.request`）：自动注入 `Authorization`、自动 traceId、超时 15s、5xx 自动重试 1 次（仅 GET 重试）
- **登录态续期**：401 自动触发 `uni.login` → 后端换取新 token → 重放原请求（队列化避免并发刷新，详见模板 `src/utils/request.ts`）
- **请求合并**：商品列表分页加载使用游标分页；详情页用 `Promise.all` 并行发起「商品详情 + 评价摘要 + 推荐」三请求
- **离线兜底**：购物车操作失败时本地写入 pending 队列，网络恢复（监听 `uni.onNetworkStatusChange`）后批量上行
- **多端兼容**：H5 端通过 vite 代理避开跨域；小程序端走真实域名，需在公众平台配置 `request 合法域名`

### 3.6 性能优化要点

| 项目 | 措施 |
|---|---|
| 首屏 | 骨架屏组件 + `onLaunch` 预拉关键数据 + 图片 `lazy-load` 属性 |
| 包体 | Vite tree-shaking、`drop_console`、图片 WebP、字体外链、按需引入 uni-ui、UnoCSS 替代部分 SCSS |
| 渲染 | Vue 3 响应式自动批量更新（`setData` 由 uni-app 编译层合并）；长列表用 `<scroll-view>` + 虚拟列表组件；避免在模板中放复杂表达式 |
| 网络 | CDN 缓存静态资源、HTTP 缓存头、合并请求；`uni.request` 默认开启 keep-alive |
| 启动 | 主包仅留 tabBar 与启动必备；商品/订单/营销 三个分包 + `preloadRule` 预下载 |
| 调试 | dev 注入 vConsole（H5 端）/ 微信开发者工具 perf 面板；构建产物 `--report` 看体积构成 |

## 4. 后端架构

### 4.1 模块化单体（Monolith-Modular）

按 Go 社区 **standard project layout** 组织代码：业务领域放 `internal/modules/<domain>/`，每个领域包内自带 `handler / service / repository / dto / model`，对外通过 service 层接口暴露。模块间**只允许通过 service interface 调用，禁止跨模块直接访问 repository 或 model**——这条规则是日后能拆分微服务的关键。

依赖注入用 **Google Wire**（编译期生成代码，零运行时反射开销）。HTTP 路由用 **Gin**，每个模块自己的 `router.go` 注册到根 router group。

```
shop-server/
├── cmd/
│   ├── api/                      # API 服务入口（main.go）
│   ├── worker/                   # 异步消费者入口
│   └── job/                      # 定时任务入口
├── internal/
│   ├── app/                      # 应用初始化、Wire 注入
│   │   ├── wire.go               # Wire ProviderSet
│   │   └── server.go             # Gin engine 装配
│   ├── pkg/                      # 项目内通用组件
│   │   ├── middleware/           # JWT / 限流 / 日志 / TraceID / Recovery
│   │   ├── response/             # 统一响应包装 + 错误码
│   │   ├── errs/                 # BizError 业务异常
│   │   ├── validator/            # 自定义 validator (基于 go-playground/validator)
│   │   └── ctxutil/              # context.Context 工具 (CurrentUser, TraceID)
│   ├── infra/                    # 基础设施
│   │   ├── db/                   # GORM 初始化、事务封装
│   │   ├── cache/                # go-redis 封装
│   │   ├── oss/                  # 阿里云 OSS 封装（含 STS 临时凭证签发）
│   │   ├── wechat/               # 微信开放接口封装
│   │   └── wxpay/                # 微信支付 V3 SDK 封装
│   └── modules/
│       ├── user/                 # 用户领域
│       │   ├── handler/          # Gin Handler (HTTP 入参出参)
│       │   ├── service/          # 业务逻辑 + 事务编排
│       │   ├── repository/       # GORM 数据访问
│       │   ├── model/            # GORM 实体
│       │   ├── dto/              # 请求/响应结构体
│       │   └── router.go         # 模块路由注册
│       ├── product/              # 商品领域
│       ├── cart/                 # 购物车领域
│       ├── order/                # 订单领域 ⭐ 核心
│       ├── payment/              # 支付领域 ⭐ 核心
│       ├── inventory/            # 库存领域 ⭐ 核心 (Redis Lua)
│       ├── marketing/            # 营销（券、秒杀、拼团）
│       ├── aftersale/            # 售后
│       ├── address/              # 收货地址
│       ├── notification/         # 订阅消息/短信
│       └── admin/                # 后台管理（独立鉴权 + 路由）
├── api/                          # OpenAPI / Proto 定义（如有）
├── configs/                      # 配置文件 (yaml)
├── deployments/                  # Dockerfile / K8s manifests
├── scripts/                      # 数据库迁移、运维脚本
├── go.mod
├── go.sum
└── Makefile
```

> **关键点**：每个 `modules/<x>/service` 暴露 `Service interface`，跨模块依赖 interface 而非具体实现，便于 mock 测试与日后拆服务。例如 `order/service` 里持有 `inventory.Service` 的接口而不是 `*inventoryService` 实例。

### 4.2 模块依赖规则

```
admin ──┐
        ↓
order ──→ payment ──→ wechat-pay
  │  ↘
  │   inventory ──→ redis
  │
  ↓
product ──→ cos / cdn
  │
  ↓
cart ──→ marketing
  │
  ↓
user ──→ wechat-auth
```

**核心规则**：
- 上层模块可以依赖下层，下层不得反向依赖。
- `order` 是中枢模块，编排 `payment` / `inventory` / `marketing` / `notification`。
- 所有跨模块调用走 `*.service.ts` 暴露的方法，**禁止直接 inject 别的模块的 Repository**。

### 4.3 接口设计规范

- **协议**：HTTPS + JSON。统一前缀 `/api/v1/`。
- **风格**：RESTful 为主，复杂操作（下单、支付回调）用 RPC 风格 POST。
- **响应包装**：

```json
{
  "code": 0,
  "message": "ok",
  "data": { ... },
  "traceId": "abc123"
}
```

- **错误码**：四段式 `<HTTP><模块><错误>`，例如 `40010001` = 400 + 用户模块 + 用户名非法。
- **鉴权**：JWT（HS256），payload 含 `uid`、`openid`、`exp`；access token 2h，refresh token 30d；放在 `Authorization: Bearer xxx`。
- **幂等**：所有写接口必须支持 `Idempotency-Key` 头（订单创建、支付回调强制要求）。
- **分页**：游标分页（`cursor` + `limit`），不用 offset 避免深度分页。
- **版本**：路径版本 `/v1/`、`/v2/`，破坏性变更才升版本。

### 4.4 关键中间件 & 横切关注点

Gin 中间件链按注册顺序依次执行，**Recovery → TraceID → Logger → CORS → RateLimit → JWTAuth → 业务 Handler**。

| 关注点 | Go 生态实现 |
|---|---|
| HTTP 框架 | **Gin** (`github.com/gin-gonic/gin`) — 路由、绑定、中间件 |
| 依赖注入 | **Wire** (`github.com/google/wire`) — 编译期生成，零反射开销 |
| ORM | **GORM v2** (`gorm.io/gorm`) — 复杂查询走原生 SQL + sqlx 兜底 |
| 配置 | **Viper** (`github.com/spf13/viper`) — yaml + env，支持热加载 |
| 鉴权 | **golang-jwt/jwt v5** + 自研 `JWTAuth()` 中间件，公开接口用路由分组跳过 |
| 限流 | **uber-go/ratelimit**（单机）+ Redis 滑动窗口（分布式）；登录/下单/秒杀分通道配额 |
| 日志 | **zap** (`go.uber.org/zap`) + 全局 LoggerMiddleware；JSON 结构化输出，自动注入 traceId / userId |
| 异常恢复 | Gin `Recovery()` + 自定义 `errs.BizError`，统一在 `ErrorHandler` 中间件转 HTTP 响应 |
| 参数校验 | Gin 内置 `binding:"required,min=1"` + `go-playground/validator/v10` 自定义规则；返回字段级错误 |
| 上下文传递 | 标准 `context.Context` 透传 traceId / userId / DB 事务句柄 |
| 链路追踪 | **OpenTelemetry-Go** SDK（MVP 阶段先在日志中接入 traceId，后期接 Jaeger / 阿里云 ARMS） |
| 优雅关停 | `http.Server.Shutdown` + `signal.NotifyContext`，配合 K8s preStop hook |
| 异步任务 | **asynq** (`github.com/hibiken/asynq`) 基于 Redis 的任务队列；MVP 阶段直接用 `domain_event` 表 + cron worker，演进期切到 asynq / RocketMQ |
| 定时任务 | **robfig/cron v3**，运行在独立的 `cmd/job` 二进制 |
| 单元测试 | `testify` + `gomock`（基于 service interface 生成 mock） |
| 接口文档 | `swaggo/swag` 注解生成 OpenAPI |

## 5. 数据架构

### 5.1 存储选型

| 数据类型 | 存储 | 说明 |
|---|---|---|
| 业务主数据 | **MySQL 8.0** (RDS) | 用户、商品、订单、支付、库存等强一致数据 |
| 热点缓存 | **Redis 6.x** | 商品详情缓存、购物车、库存扣减、限流、Session、验证码 |
| 图片/视频 | **阿里云 OSS** + CDN | 小程序通过 STS 临时凭证 / 签名 URL 上传，前台通过 CDN 拉取 |
| 日志 | **本地文件 + 阿里云 SLS** | MVP 用本地 + SLS 日志服务；演进到 ELK |
| 搜索 | MySQL LIKE（MVP）→ **Elasticsearch**（演进） | 商品量 < 5 万时 MySQL 足够 |
| 数据分析 | MySQL 从库 + 手工查询（MVP）→ **数仓**（演进） | |

### 5.2 数据建模强制规范：逻辑删除（Soft Delete）⭐

**所有业务表禁止物理删除（DELETE），统一采用逻辑删除。**

理由：
1. **可追溯**：电商业务订单、商品、用户的删除都涉及对账、审计、退款等链路，物理删除会导致历史关联数据失效。
2. **可恢复**：用户/运营误操作可一键恢复。
3. **合规**：用户注销、商品下架、营销活动结束等都需要保留历史数据满足监管要求。

**统一实现规范**：

| 项 | 约定 |
|---|---|
| 字段 | 所有表必须包含 `deleted_at TIMESTAMP NULL DEFAULT NULL` |
| 含义 | `deleted_at IS NULL` 表示有效；非 NULL 即被删除，值为删除时间 |
| GORM 类型 | `DeletedAt gorm.DeletedAt \`gorm:"column:deleted_at;index"\``，GORM v2 会自动在 `Find/Update/Delete` 时追加 `deleted_at IS NULL` 过滤 |
| 索引 | 所有查询索引最左/最右带上 `deleted_at`，或建立部分索引（仅 deleted_at IS NULL） |
| 唯一约束 | 不能直接 UNIQUE，必须 `UNIQUE (xxx, deleted_at)`——否则同一业务键无法重新创建 |
| 删除 SQL | 由 GORM `Delete()` 自动转换为 `UPDATE xxx SET deleted_at=NOW()`；禁止 raw SQL 写 `DELETE FROM` |
| 联表查询 | JOIN 时主从表都要带 `deleted_at IS NULL`（GORM Preload 已自动处理） |
| 后台恢复 | 提供 `Unscoped().Update("deleted_at", nil)` 接口供运营恢复 |
| 真正物理清理 | 仅限定时任务清理 deleted_at > 18 个月的数据，单独审批；订单/支付永不物理删 |

**BaseModel 统一封装**（所有 model 嵌入）：

```go
// internal/pkg/model/base.go
type BaseModel struct {
    ID        uint64         `gorm:"primaryKey;autoIncrement" json:"id"`
    CreatedAt time.Time      `gorm:"column:created_at;not null" json:"createdAt"`
    UpdatedAt time.Time      `gorm:"column:updated_at;not null" json:"updatedAt"`
    DeletedAt gorm.DeletedAt `gorm:"column:deleted_at;index" json:"-"`
}
```

> **唯一索引示例**：原来 `UNIQUE (openid)`，必须改为 `UNIQUE (openid, deleted_at)` 或 `UNIQUE INDEX uk_openid (openid) WHERE deleted_at IS NULL`（PostgreSQL 支持，MySQL 用前者）。

### 5.4 索引与分库分表策略

- MVP 阶段全部单库单表，常用查询条件覆盖索引（如 `order(user_id, status, created_at, deleted_at)`）。
- 所有索引必须把 `deleted_at` 作为最右列纳入，避免逻辑删后仍走索引扫描。
- 历史订单（>180 天）通过定时任务归档到 `order_archive` 表，主表保持轻量；归档同样是写入新表 + 标记 `deleted_at`，**不物理删除原表**。
- 物理清理：仅由独立定时任务清理 `deleted_at < now()-18个月` 且无关联引用的数据，订单/支付/库存日志永久保留。
- 分库分表预留：`order` 按 `user_id` 哈希预留分片字段（即使现在不分），便于后续 ShardingSphere 接入。

### 5.5 缓存策略

| 数据 | 模式 | TTL | 失效策略 |
|---|---|---|---|
| 商品详情 | Cache-Aside | 30 min | 商品后台更新时主动 delete |
| 商品列表（分类/首页） | Cache-Aside | 5 min | TTL 过期 + 后台更新主动失效 |
| 用户信息 | Cache-Aside | 1 h | 修改时主动失效 |
| 购物车 | Write-Through | 永久 | 与 DB 双写，DB 兜底 |
| 库存 | **Redis 为权威源**，DB 异步落账 | 永久 | 见 §6.3 |
| Session/Token | Redis | 2 h | 自然过期 |
| 限流计数 | Redis | 滑动窗口 | 自然过期 |

**防三类问题**：
- **缓存穿透**：不存在的 key 写入空对象（短 TTL 60s）+ 布隆过滤器（可选）。
- **缓存击穿**：热点 key 加 Redis 互斥锁（SETNX），单实例回源。
- **缓存雪崩**：TTL 加随机抖动 ± 10%，关键缓存预热。

## 6. 关键业务流程设计

### 6.1 登录与会话

1. 小程序调用 `wx.login()` 拿到 `code`。
2. 前端 `POST /api/v1/auth/login {code, encryptedData?, iv?}`。
3. 后端 `code2session` → 拿 `openid` + `session_key`。
4. 找/建 `user` 记录，签发 `accessToken (JWT, 2h)` + `refreshToken (30d)`。
5. 返回 `{token, user, isNew}`。
6. 客户端存储 token，后续请求带 `Authorization`。

> 微信手机号授权：用户点击「微信号绑定手机号」按钮触发 `getPhoneNumber`，后端用 `code` 调用微信 API 解密拿到手机号，更新 `user.phone`。

### 6.2 下单流程（核心）

```
[用户点确认下单] 
   │
   ▼
1. 校验：登录态 / 收货地址 / SKU 上架 / 规格价格 / 优惠券可用
   │
   ▼
2. 价格试算（trade preview）：商品金额 + 运费 - 优惠 = 应付
   │   ←── 客户端拿到 preview 结果展示给用户
   ▼
3. [用户确认支付] → POST /api/v1/orders   (Idempotency-Key 必填)
   │
   ▼
4. 事务开启
   ├─ 4.1 锁定库存（Redis Lua 原子扣减 available→locked，写 inventory_log）
   ├─ 4.2 生成订单（order + order_item，状态 = WAIT_PAY）
   ├─ 4.3 锁定优惠券（user_coupon.status = LOCKED）
   ├─ 4.4 写 domain_event(topic=ORDER_CREATED)
   └─ 提交事务
   │
   ▼
5. 调微信支付统一下单 → 拿到 prepay_id 与签名参数
   │
   ▼
6. 返回客户端，客户端调 wx.requestPayment 拉起支付
```

**库存扣减关键点**：用 Redis Lua 脚本保证原子性（先 check 后 deduct），避免超卖；DB 库存通过 `inventory_log` 异步对账修正，**Redis 是权威源**。

### 6.3 支付回调与状态机

订单状态机：

```
WAIT_PAY ──支付成功──→ PAID ──发货──→ SHIPPED ──确认收货──→ FINISHED
   │                     │
   │                     └──退款──→ REFUNDING ──→ REFUNDED
   │
   └──超时(30min)/取消──→ CANCELED
```

**支付回调（关键幂等设计）**：

```
微信支付服务器 ──→ /api/v1/pay/wechat/notify
   │
   ▼
1. 验签（微信 V3 SHA256-RSA）
2. 用 transaction_id 加 Redis 分布式锁（10s）
3. SELECT payment_record WHERE transaction_id = ? FOR UPDATE
   ├─ 已 SUCCESS → 直接返回 success（幂等）
   └─ 未处理：
       ├─ UPDATE payment_record SET status=SUCCESS
       ├─ UPDATE order SET status=PAID, pay_time=now()
       ├─ INVENTORY locked → deducted
       ├─ user_coupon LOCKED → USED
       └─ 写 domain_event(topic=ORDER_PAID)
4. 返回 success
```

`domain_event(ORDER_PAID)` 由独立 worker 消费，触发：模板消息推送、积分发放、销量统计刷新等**非关键路径**。

### 6.4 超时关单

定时任务每分钟扫描 `WAIT_PAY` 且 `created_at < now()-30min` 的订单：
1. 主动调微信支付查单 API 兜底（避免回调丢失）。
2. 如确未支付：关闭订单，释放库存（`inventory_log` 写入 RELEASE），解锁优惠券。

### 6.5 售后流程

```
用户发起退款/退货 → after_sale_apply (PENDING)
  → 商家审核
    ├─ 拒绝 → REJECTED
    └─ 通过 → 微信退款 API → REFUNDING → 微信退款回调 → REFUNDED
```

退款必须**原路退回**，不允许改路径；退款到账以**微信回调**为准。

## 7. 安全设计

### 7.1 身份与鉴权

- JWT 短期 access token + 长期 refresh token；refresh token 落 Redis 支持单点强制下线。
- 关键操作（修改手机号、退款、提现）二次验证（短信验证码 / 支付密码 / 微信弹窗）。
- 后台管理使用独立账号体系，RBAC 模型（用户-角色-权限），后台域名独立。

### 7.2 接口安全（待定）

- **传输**：全站 HTTPS，TLS 1.2+，HSTS。
- **签名**：开放给第三方/H5 的接口要求时间戳 + nonce + 签名，防重放。
- **限流**：
  - 全局：单 IP 100 req/min。
  - 登录：单 IP 10 次/min，单 openid 5 次/min。
  - 下单：单用户 10 单/min。
  - 秒杀：令牌桶 + 排队。
- **越权**：所有查询必须带 user_id 维度过滤；后台用对象级权限校验（不能直接用 id 跳转）。

### 7.3 数据安全（待定）

- **PII 加密**：手机号、收货地址明细字段在 DB 中 AES-256 
- **敏感日志脱敏**：日志中手机号 `138****1234`、身份证打码、token 不落日志。
- **支付信息**：商户号 / API 密钥 / 证书放运维金库（如 Vault），不入代码仓库。
- **数据备份**：MySQL 全量日备 + binlog 实时；备份保留 30 天，每月演练恢复一次。

### 7.4 微信支付专项

- 商户证书与 APIv3 密钥分环境隔离；测试环境不得使用生产密钥。
- 回调接口必须验签；任何参数信任前先校验签名。
- 金额必须**以分为单位整数**，避免浮点。
- 退款单号、订单号、商户单号、微信单号一一对应，账务可追。

### 7.5 风控（待定）

- 设备指纹 + openid + IP 三维度记录；异常登录（异地、短时间多 openid）触发风控拦截。
- 优惠券领取、秒杀下单按用户 + 设备 + IP 维度限频。
- 高危操作（大额退款、批量改价）走二次审批 + 全量审计日志。

  1. push → `go vet` + `golangci-lint` + `go test ./...` + 覆盖率
  2. 合并 main → `CGO_ENABLED=0 go build -ldflags="-s -w"` 构建静态二进制 → 打 Docker 镜像（基于 `alpine` ~15MB） → 推到阿里云 ACR
  3. test 环境自动发布（ACK rollout / systemd reload）
  4. staging 自动 + 人工触发到 prod
  5. prod 蓝绿/滚动发布，支持一键回滚（前一个镜像 tag 保留 30 天）
- 小程序发布：体验版二维码 → 测试 → 提审 → 全量/灰度（微信原生灰度比例）。
- **数据库变更**：所有 DDL 走 `golang-migrate/migrate`（与 Go 生态契合，CI 内置执行）；线上 DDL 必须 review，避免锁表。

## 9. 可观测性（日志 / 监控 / 告警）

### 9.1 日志

- 应用日志：JSON 结构化（zap），字段含 `traceId / userId / module / level / msg`，自动落本地 + 阿里云 SLS。
- 访问日志：Nginx 标准格式 + 响应时间 + traceId。
- 审计日志：后台所有写操作记录到独立 `audit_log` 表，180 天归档。

### 9.2 监控指标（黄金信号）

| 层 | 关键指标 |
|---|---|
| 业务 | 订单数/h、支付成功率、客单价、首页 PV/UV、转化漏斗（曝光→详情→加购→下单→支付） |
| 应用 | QPS、P50/P95/P99、错误率、慢查询、线程/事件循环延迟 |
| 中间件 | Redis 命中率、连接数；MySQL 连接数、慢日志、复制延迟 |
| 主机 | CPU/内存/磁盘/网络/负载 |

可视化：阿里云云监控 + ARMS 大盘（MVP）→ Grafana + Prometheus（演进）。

### 9.3 告警

分级管理（P0 立即电话、P1 5 分钟内 IM、P2 邮件）：
- P0：服务不可用、支付成功率 < 95%、数据库主从延迟 > 60s、订单异常激增
- P1：P99 > 1s 持续 5min、错误率 > 1%、Redis 命中率 < 90%
- P2：磁盘使用率 > 80%、CPU > 70% 持续 30min

### 9.4 全链路追踪

OpenTelemetry SDK 采集，TraceId 在「网关 → 服务 → 数据库 / 第三方」全链路透传；客户端埋点也带上同一 traceId，**便于线上问题用一个 traceId 定位完整调用链**。

## 13. 附录

### 13.1 命名规范

- 仓库：`shop-miniprogram`、`shop-server`、`shop-admin-web`
- Go module：`github.com/<org>/shop-server`，业务包名 `<module>`（如 `order`、`payment`），不带前缀
- 表名：单数小写下划线，如 `order_item`
- 字段：`snake_case`；时间字段统一 `created_at` / `updated_at` / `deleted_at`（GORM Model 用 `gorm.Model` 或自定义 BaseModel）
- 结构体：导出字段使用 PascalCase + GORM/JSON tag，例如 `OrderNo string \`gorm:"column:order_no;uniqueIndex" json:"orderNo"\``
- 接口：`/api/v1/<resource>/<sub-resource>` 复数
